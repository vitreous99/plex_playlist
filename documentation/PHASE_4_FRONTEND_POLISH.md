# Phase 4 — Frontend, Polish & Deployment

**Goal:** Provide a user-friendly interface, finalize configuration, and prepare the application for reliable daily use.

**Prerequisite:** Phase 3 complete (full backend pipeline working end-to-end via API).

---

| # | Step | Description | Definition of Done |
|---|------|-------------|-------------------|
| 4.1 | **Design the frontend layout** | Create `frontend/index.html` with the UI skeleton: a text input for the natural-language prompt, a number input for track count, a "Generate" button, a client selector dropdown, a results area to display the generated playlist, and a "Play" / "Save Playlist" button pair. Use clean, minimal CSS. | HTML file renders a usable, readable layout in a browser. |
| 4.2 | **Implement prompt submission & results display** | Add JavaScript to: POST the prompt to `/api/suggest` on "Generate" click, display the returned track list (title, artist, reasoning) in the results area. Show a loading spinner during the request. Handle and display errors. | Typing a prompt and clicking Generate populates the results area with suggested tracks. |
| 4.3 | **Implement client selector** | On page load, fetch `GET /api/clients` and populate the client dropdown. Allow the user to pick a target playback device. Show a fallback message if no clients are found. | Dropdown shows real client names from the Plex server. |
| 4.4 | **Wire up Play and Save actions** | "Play" button calls `POST /api/playlist/play` with the prompt, track count, and selected client. "Save" button calls `POST /api/playlist/save`. Show success/failure toasts in the UI. Disable buttons during processing. | Clicking Play starts music on the selected device; clicking Save creates a Plex playlist. Both show confirmation or error messages. |
| 4.5 | **Add sync management UI** | Add a "Sync Library" button that triggers `POST /api/sync`. Poll `GET /api/sync/status` every 2 seconds while in progress. Display a progress indicator and the last-synced timestamp. | User can trigger and monitor a library sync entirely from the frontend. |
| 4.6 | **Add diagnostics / status panel** | Create a status section (or modal) that calls `GET /api/diagnostics` and displays: Plex connection ✅/❌, Ollama connection ✅/❌, GPU detected ✅/❌, tracks synced count, last sync time, model loaded. | User can see full system health at a glance from the UI. |
| 4.7 | **Serve the frontend via Docker** | Update `docker-compose.yml` frontend service. Either use nginx to serve `frontend/` as static files, or mount `frontend/` into the FastAPI app and serve via `StaticFiles`. Ensure the frontend is accessible on the configured host port. | Navigating to `http://localhost:3000` (or configured port) loads the complete UI. |
| 4.8 | **Implement automatic sync on first startup** | On FastAPI app startup (lifespan event), check if the `tracks` table is empty. If so, trigger the metadata sync automatically in the background so the user doesn't need to manually sync before their first prompt. | First `docker compose up` with an empty DB triggers a background sync; user sees progress in the UI. |
| 4.9 | **Configuration validation on startup** | On app initialization, validate: all required env vars are set and non-empty, Plex URL is reachable and token is valid, Ollama is reachable, specified model is available (log a warning if not pulled yet). Log clear startup diagnostics to stdout. | Missing or invalid config causes a clear, human-readable error at startup instead of a cryptic crash. |
| 4.10 | **Write a comprehensive README** | Replace `README.md` with: project overview, architecture diagram (Mermaid), prerequisites (Docker, NVIDIA container toolkit, Plex token), step-by-step setup instructions (clone → copy `.env.example` → fill in → `docker compose up`), usage guide with screenshots or examples, troubleshooting FAQ covering common issues (GPU not detected, client not found, Shield asleep). | A brand-new user can go from zero to a running app by following the README alone. |
| 4.11 | **End-to-end acceptance testing** | Write a test script or checklist that exercises the full workflow: start stack → sync library → submit prompt → verify suggestions appear → play on client → save playlist → verify in Plex UI. Document expected results for each step. | The test script/checklist runs successfully against a live Plex + Ollama environment with all steps passing. |
| 4.12 | **Tag and release v1.0.0** | Ensure all code and docs are committed. Tag the repo as `v1.0.0`. Create a GitHub release with a changelog summarizing all features delivered across the four phases. | `git tag v1.0.0` exists; GitHub release page shows a complete feature summary. |
