**Goal:** Implement the two data-processing pillars: library metadata sync from Plex into SQLite, and the Ollama-powered semantic prompt processor.

**Prerequisite:** Phase 1 complete (Docker stack running, FastAPI app bootstrapped, SQLite schema created).

| # | Step | Description | Definition of Done |
|---|------|-------------|-------------------|
| 2.1 | **Implement Plex server connection utility** | Create `backend/app/services/plex_client.py`. On module load, connect to Plex via `PlexServer(PLEX_URL, PLEX_TOKEN)` using env vars. Expose a helper `get_music_section()` that returns the music library section. Include try/except for unreachable server or bad token with clear error messages. | Importing the module with valid creds connects successfully; with bad creds raises a descriptive error. |
| 2.2 | **Build the Metadata Sync module** | Create `backend/app/services/sync.py`. Iterate `music_section.searchTracks()` to fetch all tracks. For each track extract: `ratingKey`, `title`, `artist().title`, `album().title`, genres (joined string), styles (joined string), and sonic-analysis status. Upsert each row into the SQLite `tracks` table (insert or update on `rating_key` conflict). | Running the sync populates the SQLite DB; row count matches the Plex library track count. |
| 2.3 | **Expose sync as a FastAPI endpoint + background task** | Add `POST /api/sync` that triggers the metadata sync as a FastAPI `BackgroundTasks` job. Add `GET /api/sync/status` returning `{"synced_tracks": int, "total_tracks": int, "last_synced_at": str, "in_progress": bool}`. Store sync state in a module-level variable or simple DB row. | Calling POST starts sync in background; GET shows live progress and completion state. |
| 2.4 | **Implement SQLite query helpers** | Create `backend/app/services/library_search.py` with functions: `search_tracks_by_keywords(keywords: list[str]) -> list[Track]` (searches title, artist, genre, style), `get_distinct_artists() -> list[str]`, `get_distinct_genres() -> list[str]`, `get_tracks_by_artist(artist: str) -> list[Track]`. | 2.5 | **Define Pydantic models for LLM I/O** | Create `backend/app/models/schemas.py` with: `SuggestedTrack(title: str, artist: str, album: Optional[str], reasoning: str)`, `PlaylistResponse(name: str, description: str, tracks: List[SuggestedTrack])`, `PromptRequest(prompt: str, track_count: int = 20)`. | Models instantiate correctly; `PlaylistResponse.model_json_schema()` outputs valid JSON schema. |
| 2.6 | **Build the Prompt Processor** | Create `backend/app/services/prompt_processor.py`. Given a user prompt: (a) extract keywords via simple NLP (split, remove stopwords), (b) query SQLite for matching artists and genres as a "context pool", (c) construct a system prompt instructing the LLM to select tracks from the user's library, injecting the context pool and requiring output in `PlaylistResponse` schema. | Given "chill jazz for a rainy night", function returns a complete prompt string containing relevant artists/genres from the library. |
| 2.7 | **Implement Ollama API integration** | Create `backend/app/services/ollama_client.py`. Use the `ollama` Python library or `httpx` to call the Ollama chat endpoint. Send the constructed prompt with `format=PlaylistResponse.model_json_schema()`, `temperature=0`, and the model from `DEFAULT_MODEL` env var. Parse the JSON response into a `PlaylistResponse` Pydantic object. | Calling the function with a test prompt returns a valid, parsed `PlaylistResponse`. |
| 2.8 | **Add retry & over-request logic for LLM** | Wrap the Ollama call in a retry loop (max 3 attempts). If JSON parsing fails, retry. If the returned track count is below the requested amount, re-prompt asking for 50% more tracks. Log each attempt with the attempt number and reason for retry. | LLM returning too few tracks triggers an automatic retry; JSON parse failures are retried; all retries are logged. |
| 2.9 | **Expose prompt-to-suggestions endpoint** | Add `POST /api/suggest` accepting a `PromptRequest` body. Orchestrate: prompt processor â†’ Ollama client â†’ return `PlaylistResponse` JSON. Include error handling for Ollama down, empty results, etc. | `curl -X POST /api/suggest -H 'Content-Type: application/json' -d '{"prompt":"upbeat 90s rock","track_count":20}'` returns valid JSON. |
| 2.10 | **Write integration tests for Phase 2** | Create `backend/tests/test_sync.py`, `test_library_search.py`, `test_prompt_processor.py`, `test_ollama_client.py`, `test_api_suggest.py`.
